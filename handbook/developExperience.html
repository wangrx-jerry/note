    # clientHeight, offsetHeight, scrollHeight的区别

    • clientHeight 取值方式：document.documentElement.clientHeight 作用：获取可视区的高度（不包括滚动条 & 不包括padding宽度）
    • offsetHeight 取值方式：document.documentElement.offsetHeight 作用：获取文本的高度（不包括滚动条宽度）
    • scrollHeight 取值方式：document.documentElement.scrollHeight 作用：获取文本高度（滚动条展开的高度）

    注意： 如果之前声明了DTD（DOCTYPE）可以使用document.documentElement 如果没有声明DTD，使用document.body

    所以判断浏览器是否滚动到页面最底部的方法：document.documentElement.scrollHeight ?= window.innerHeight + element.scrollTop

    # mvc mvp mvvm联系的区别

    • 三者是目前比较流行的软件架构
    • 前端开发无非就是就是将数据库的数据渲染到用户界面上，
    • 其中m（modal）就是数据的容器，通常从数据库中映射出来，v（view）就是展现出来的用户界面
    • 这些架构所做的就是将数据渲染到界面上，但是数据库映射出来的数据往往不能直接用于界面渲染，往往需要一些处理，三个架构的不同点就是这中间的处理方式的不同
    • 其中mvvm（典型代表：vue.js, angular.js），使用viewmodal实现view和modal之间的关联，mvvm的看法是：给view里面的各种控件也定义一个对应的数据对象，这样，只要修改这个数据对象，view里面显示的内容就自动跟着刷新，而在view 里做了任何操作

    # 盒子模型

    盒子模型有两种：一种是标准模式(Standards)，另一种是怪异模式(Quirks) 区别：

    • 在W3C 标准中，如果设置一个元素的高度和宽度，指的是元素内容的宽度和高度，而在Quirks模式下，IE的宽度和高度还包含了padding和border
    • 设置行内元素的高度：在Standards模式下，给等行内元素设置wdith和Height都不会生效，而在Quirks模式下，则会生效。
    • 设置百分比高度：在Standards模式下，一个元素的高度是由包含的内容来决定的，如果父元素没有设置百分比的高度，自元素设置一个百分比的高度是无效的，
    • 用margin：0 auto设置水平居中：在Standards模式下可以让元素居中，但是在Quirks模式下是无效的 DOCTYPE都是标准的文档类型，无论使用哪种模式完整定义DOCTYPE，都会触发标准模式，而如果DOCTYPE缺失则在ie6，ie7，ie8下将会触发怪异模式（quirks 模式）。
    box-sizing: content-box;//采用标准模式解析计算
box-sizing: border-box;//采用怪异模式解析计算

    # CSS3视窗单位vw、vh、vmin、vmax 参考 (opens new window)

    vw、vh、vmin和vmax是CSS3中的新单位，是一种视窗单位，也是相对单位。它们的大小都是由视窗大小来决定的，单位1，代表类似于1%。具体描述如下：

    • vw：视窗宽度的百分比
    • vh：视窗高度的百分比
    • vmin：当前较小的vw和vh
    • vmax：当前较大的vw和vh

    视窗，指的是浏览器可视区域的宽高，也就是window.innerWidth/window.innerHeight。1vw就是1%的浏览器的宽度。100vw就是整个视窗的宽度，他和百分比的区别：%是相对父元素的大小设定的比率，vm是驶入大小决定的

    vmin和vmax 使用场景：元素的宽/高不受设备显示方向的影响（比如：移动端设备横屏和竖屏字体大小的不同可以使用vmin/vmax设置）

    # AMD和CMD

    • AMD 是 RequireJS 在推广过程中对模块定义的规范化产出。

    • CMD 是 SeaJS 在推广过程中对模块定义的规范化产出。

    • 区别：

      • 对于依赖的模块：AMD：提前执行，CMD是延迟执行
      • AMD推崇依赖前置，CMD推崇依赖就近
      // AMD
define(['./a', './b'], function(a, b) {
    // 依赖必须一开始就写好
    a.doSomething()
    // 此处略去 100 行
    b.doSomething()
    ...
})
// CMD
define(function(require, exports, module)
{
    var a = require('./a')
    a.doSomething()
    // 此处略去 100 行
    var b = require('./b')
    // 依赖可以就近书写
    b.doSomething()
    // ...
})

    • AMD 的 API 默认是一个当多个用，CMD 的 API 严格区分，推崇职责单一。比如 AMD 里，require 分全局 require 和局部 require，都叫 require。CMD 里，没有全局 require，而是根据模块系统的完备性，提供 seajs.use 来实现模块系统的加载启动。CMD 里，每个 API 都简单纯粹。

    # 访问一个网站的背后步骤参考 (opens new window)

    过程设计的一些网络协议（以百度为例）：

    1. DNS：获取ip地址。在DNS协议中，PC会向你的本地DNS服务器求助（一般是路由器），希望从本地DNS服务器那里得到百度的IP，得到就好，得不到还得向更高层次的DNS服务器求助，最终总能得到百度的IP；
    2. tcp协议：请求向服务器发送数据。在TCP协议中，建立TCP需要与百度服务器握手三次，你先告诉服务器你要给服务器发东西（SYN），服务器应答你并告诉你它也要给你发东西（SYN、ACK），然后你应答服务器（ACK），总共来回了3次，称为3次握手。
    3. IP协议：指定出发地（你的pc）和目的地（服务器）
    4. ARP协议：找到所有路径
    5. OSPF协议：找到最优路径请求资源 概述：DNS解析获取到ip地址 -> 建立tcp连接 -> 发送http请求 -> server接受http请求，并处理请求返回数据 -> 客户端接受数据并处理数据（渲染页面，执行js）

    补充：tcp的三次握手和四次挥手： 三次握手（连接）：

    三次握手：

    • 第一次：客户端向服务端发送连接请求
    • 第二次：服务端段接受到连接请求，向客户端发送请求应答结果
    • 第三次：客户端受到请求应答结果，如果允许请求则发送报文内容

    四次挥手：

    • 第一次：客户端发送断开连接的请求
    • 第二次：服务端接受到请求，向客户端发送消息告知客户端接受到请求
    • 第三次：服务端处理好所有数据，并发送完所有数据后告知客户端，连接可以断开了
    • 第四次：客户端接受到断开通知回复，过2MSL后再向服务端发送请求用来确定这个连接是不是真的已经被断开了

    # 减少js对性能的影响

    1. 将所有的script标签放到页面底部，也就是body闭合标签之前，这能确保在脚本执行前页面已经完成了DOM树渲染。
    2. 尽可能地合并脚本。页面中的script标签越少，加载也就越快，响应也越迅速。无论是外链脚本还是内嵌脚本都是如此。
    3. 采用无阻塞下载 JavaScript 脚本的方法：
      1. 使用script标签的 defer 属性（仅适用于 IE 和 Firefox 3.5 以上版本）；
      2. 使用动态创建的script元素来下载并执行代码；

    # 页面重绘和重排参考 (opens new window)

    1. Reflow（回流/重排）：当它发现了某个部分发生了变化影响了布局，渲染树需要重新计算。
    2. Repaint（重绘）：改变了某个元素的背景颜色，文字颜色等，不影响元素周围或内部布局的属性。

    Reflow要比Repaint更花费时间，也就更影响性能。所以在写代码的时候，要尽量避免过多的Reflow

    reflow 的原因

    1. 页面初始化的时候
2. 操作dom的时候
3. 元素尺寸变化
4. 如果css属性发生变化
5. 浏览器的窗口尺寸变化（因为回流是根据视口的大小来计算元素的位置和大小的）

    减少回流和重绘的方法（虽然新浏览器会对此进行优化但是老版本的浏览器还是不行），例如：

    const el = document.getElementById('test');
el.style.padding = '5px';
el.style.borderLeft = '1px';
el.style.borderRight = '2px';

    有三个样式属性被修改了，每一个都会影响元素的几何结构，引起回流。当然，大部分现代浏览器都对其做了优化，因此，只会触发一次重排。但是如果在旧版的浏览器或者在上面代码执行的时候，有其他代码访问了布局信息(上文中的会触发回流的布局信息)，那么就会导致三次重排。

    我们可以合并所有的改变然后依次处理，比如我们可以采取以下的方式：

    • 使用cssText
    const el = document.getElementById('test');
el.style.cssText += 'border-left: 1px; border-right: 2px; padding: 5px;';
    • 修改css的class
    const el = document.getElementById('test');
el.class += ' className';

    # js阻塞页面渲染的的原因

    原因：js线程 和 控制生成DOM树的进程（GUI）是互斥的

    浏览器进程介绍(GUI渲染线程, JS引擎线程, 定时器触发线程, 事件触发线程, 异步HTTP请求线程)：

    • GUI线程负责页面渲染（解析HTML，CSS，构建DOM树，布局和绘制等），而该线程是与JS引擎线程互斥的，当执行JS引擎线程时，GUI渲染线程将会被挂起(冻结)
    • 定时器线程： 负责执行定时器一类的线程，当主线程依次执行时遇到定时器，会将定时器交给该线程处理，该线程将计时完毕后的事件加入到事件队列的尾部，等待JS引擎线程的执行
    • 宏任务包括：全局script任务、setTimeout、setInterval、setImmediate、I/O操作、UI rendering
    • 微任务包括：new Promise.then()、MutationObserver(HTML5新特性)等

    任务执行顺序：先执行宏任务再执行微任务，比如：

    console.log('golb1');

setTimeout(function () {
    console.log('timeout1');
    new Promise(function (resolve) {
        console.log('timeout1 promise');
        resolve();
    }).then(function () {
        console.log('timeout1 then')
    })
})

setTimeout(() => {
    console.log('timeout2')
}, 1000);

new Promise((resolve, reject) => {
    console.log('promise1')
    resolve();
}).then(res => {
    new Promise(resolve => {
        console.log('promise2');
        resolve()
    }).then(() => {
        console.log('promise2 then');
    })
    console.log('promise res')
})

    执行顺序：golb1 -> promise1 -> promise2 -> promise res -> promise2 then -> timeout1 -> timeout1-promise -> timeout1 then -> timeout2 执行顺序分析：先执行宏任务 script， 再执行微任务（插队执行：如果当前任务中包含微任务，优先执行这个微任务，宏任务则是工具插入顺序依次执行）

    # 网页渲染流程

    参考 (opens new window)

    1. 浏览器向服务器获取到html文档经过 dom解析 生成dom tree（document object modal 文档对象模型）
    2. 样式被载入和解析，构成CSSOM（css object modal 样式层叠对象模型）
    3. 在DOM和CSSDOM基础之上创建渲染树 render tree，渲染出页面
    4. 其中js是加载即执行的，同时会阻塞后面的内容，因为浏览器需要一个稳定的dom树结构，而js中很有可能有代码直接改变了dom树结构 tips:
    • 外部样式会阻塞内部脚本的执行(因为js可能会修改css和dom，所以浏览器先让页面绘制完成再执行js)。
    • 外部样式与外部脚本并行加载，但外部样式会阻塞外部脚本执行。
    • 如果外部脚本带有async属性，则外部脚本的加载与执行不受外部样式影响
    • 如果link标签是动态创建（js生成），不管有无async属性，都不会阻塞外部脚本的加载与执行。
    • DOMContentLoaded:当页面的内容解析完成后，则触发该事件,它的加载顺序：
      1. 如果页面中同时存在css和js，并且存在js在css后面，则DOMContentLoaded事件会在css加载完后才执行。
      2. 其他情况下，DOMContentLoaded都不会等待css加载，并且DOMContentLoaded事件也不会等待图片、视频等其他资源加载。

    # 构造函数，实例 参考

    function xiaoming (){
    this.name = 'xiaoming';
    this.age = 12;
}
xiaoming.protoType.sex = 'man';
var person = new xiaoming();

    当使用new操作符调用构造函数xiaoming时，会经历以下步骤：

    1. 创建一个空对象，作为将要返回的实例对象
    2. 将空对象的原型指向构造函数的prototype属性
    3. 将空对象赋值给构造函数内部的this关键字，也就是关键字会指向实例对象
    4. 开始执行构造函数内部代码

    所以上面的🌰指向关系是：

    1. person指向xiaoming.protoType
    2. xiaoming.protoType 指向 Function.protoType
    3. Function.protoType 指向 Object.protoType
